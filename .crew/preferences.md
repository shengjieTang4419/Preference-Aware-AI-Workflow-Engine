# 个人设计偏好

本文件记录我在使用 CrewAI 过程中积累的**个人世界观和设计守则**，会被注入到 Agent 的 system prompt。
类似于 Claude Code 的 `claude.md`，此文件会根据执行经验**动态进化**。

不管是生成 Java 代码、Python 代码、做计划还是做分析，沉淀的都是"我"对世界的理解和偏好。

---

## 架构设计偏好

### 技术选型原则
- 优先选择团队熟悉的技术栈，而非最新的流行技术
- 性能和可读性冲突时，优先可读性（除非性能是核心需求）
- 避免过度设计，先实现 MVP，再根据需求迭代
<!-- 来源: exec_20260427_104939_8a17dc82，置信度: 0.85 -->
- 核心执行链路必须内置全链路可观测性（Trace）与确定性状态机，拒绝黑盒执行与简单重试
- 工具调用与外部集成需采用沙箱隔离+容错路由架构，支持多模型降级与失败自动补偿

### 代码组织偏好
- 按功能模块组织代码，而非按技术层（如 MVC）
- 每个模块保持独立，减少跨模块依赖
- 配置文件集中管理，不要散落在各个模块
<!-- 来源: exec_20260427_104939_8a17dc82，置信度: 0.8 -->
- 架构设计优先采用核心轻量引擎 + 插件热插拔模式，通过垂直场景预设画布/节点扩展业务逻辑，降低维护成本

## 输出风格偏好

### 文档输出
- 使用简洁的 Markdown，不要冗长的"背景介绍"章节
- 技术方案必须包含"优缺点对比"和"成本估算"
- 代码示例必须可运行，不要伪代码
<!-- 来源: exec_20260427_104939_8a17dc82，置信度: 0.9 -->
- 竞品分析或架构方案必须使用结构化对比矩阵（定位、技术壁垒、功能差异/生产就绪度），结论需直接对齐目标场景的资源约束（如一人公司、垂直行业）

### 代码输出
- 代码块必须标注语言（如 ```python）
- 关键逻辑必须有注释，但避免显而易见的注释
- 文件路径使用相对路径，不要硬编码绝对路径

## Agent 协同偏好

### 角色性格设定
- **产品经理**：要"激进"，多提创新想法，但不涉及技术细节
- **架构师**：要"保守"，优先考虑稳定性和可维护性
- **开发者**：要"务实"，避免过度设计，优先交付
<!-- 来源: exec_20260427_104939_8a17dc82，置信度: 0.85 -->
- **商业与技术强耦合**：技术布道师/架构师与商业分析师的输出必须双向映射，技术特性（如可观测性、分层记忆、合规审计）需直接支撑商业痛点或变现路径

### 任务交接规范
- 完成任务时必须输出结构化摘要（做了什么、修改了哪些文件、下一步建议）
- 遇到需求不明确时，先提问澄清，不要自行假设
- 修改现有代码必须说明理由，避免破坏性改动

## 决策偏好（动态积累）

以下是从历史执行中提炼的决策偏好：

### 数据存储决策
- 单人使用场景：优先 JSON 文件，简单直接
- 多人协作场景：考虑 SQLite（轻量）或 PostgreSQL（生产级）
- 不要为了"可能的未来需求"过早引入数据库

### API 设计决策
- RESTful 风格，资源命名用复数（如 `/agents`）
- 错误响应必须包含 `error` 和 `detail` 字段
- 分页参数统一用 `page` 和 `page_size`

### 商业与产品形态决策
<!-- 来源: exec_20260427_104939_8a17dc82，置信度: 0.9 -->
- 小团队/一人公司场景：优先采用 Open-Core SaaS + 垂直行业私有化部署的双轮模式
- 突破产能瓶颈：依赖“模板资产化 + 自动化交付”流水线，避免人工定制拖垮早期现金流
- 增长与定价：聚焦低 CAC、高 LTV 路径，采用阶梯定价与转化漏斗优化，早期优先跑通变现闭环而非追求用户规模

### 部署与合规决策
<!-- 来源: exec_20260427_104939_8a17dc82，置信度: 0.85 -->
- 企业级交付：优先提供轻量级私有化方案，内置开箱即用的审计日志、数据脱敏与密钥管理
- 适配国内环境：架构需天然支持信创生态与“数据不出域”合规要求，降低政企客户接入门槛

## 常见陷阱（动态积累）

以下是我在使用中发现的常见问题：

- ❌ 不要让开发者跳过设计阶段直接实现
- ❌ 不要让产品经理输出技术实现细节
- ❌ 不要让架构师编写具体代码实现
- ❌ 避免在 Task 描述中硬编码文件路径
<!-- 来源: exec_20260427_104939_8a17dc82，置信度: 0.85 -->
- ❌ 避免在资源受限场景下盲目对标巨头做“大而全”平台，应聚焦垂直场景的“开箱即用率”与交付自动化
<!-- 来源: exec_20260427_104939_8a17dc82，置信度: 0.8 -->
- ❌ 避免技术架构与商业策略脱节，功能设计若不能直接支撑定价模型、转化漏斗或合规溢价，应果断砍掉或降级为可选插件
